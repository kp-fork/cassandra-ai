## 2. 가장 심각한 발견: Phase 1 마이그레이션 미적용

> 커밋 메시지는 "Phase 1 완료"이나, 운영 DB에는 제약이 없음.

### 현황

| 항목 | 상태 |
|---|---|
| `schema.prisma`에 `isCurrent` 필드 | ✅ 추가됨 (`schema.prisma:89`) |
| `schema.prisma`에 `@@unique([corpId, personId, role])` | ✅ 추가됨 (`schema.prisma:94`) |
| `prisma/migrations/`에 Phase 1 마이그레이션 폴더 | ❌ **없음** |
| 마지막 마이그레이션 | `20260611160045_add_batch_job` (Phase 1 이전) |

### 영향

`prisma migrate dev`가 실행되지 않았으므로, **운영 DB(Neon)에는 `isCurrent` 컬럼과 unique 제약이 존재하지 않음.**

- `fetch-officers.ts`의 `upsert(where: { corpId_personId_role })` → **실행 시 런타임 에러** (제약 없음)
- 또는 제약 없이 동작하더라도 동일 관계 **무한 중복 생성**

즉, 관계망의 핵심인 "임원·주요주주 → 회사" 엣지가 DB에 들어가지 않는 **근본 원인이 해결된 것이 아님.**

### 권장 조치

```bash
npx prisma migrate dev --name fix_relation_constraints
npx prisma generate
# 운영 DB 반영 확인:
#   \d "CorpPersonRelation"  → isCurrent 컬럼, unique 제약 존재 여부
```

## 5. 결론

리팩토링은 **코드 작성 측면에서는 상당히 진행**되었으나(특히 STAGE 0/1 핫픽스는 실제로 잘 적용됨), **"작성 = 동작"이라는 착각**이 위험한 시점.

- **Phase 1** 은 스키마만 고치고 DB엔 반영 안 됨 → 관계망 데이터가 들어갈 구조적 토대가 없음
- **Phase 2/3** 은 스크립트만 있고 실행 증거가 없음 → 실제로 Fund·flags가 생성되는지 불명
- **이슈 I/H** 는 리스크 분석과 캐시 신선도를 좌우하는 단순 버그인데 미해결

다음 단계는 **"새 코드를 더 쓰는 것"이 아니라 "기존 코드가 동작하는지 증명하는 것"** 이어야 함. 위 권장사항 1~3부터 순차 적용을 제안.

### ❌ 틀린 주장 2개

#### 1. "Phase 1 마이그레이션 미적용 — 운영 DB에 제약 없음"

**사실과 다름.** DB를 직접 조회한 결과:

전체 CorpPersonRelation 수: 2997
